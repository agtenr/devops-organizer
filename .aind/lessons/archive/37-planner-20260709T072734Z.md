## Observation
I planned an azuredevops@microsoft.com native-sender gate (from AC §1 and categorization-domain.md, which both frame the ADO sender as the signal that decides whether the rules apply). The reviewer rejected the gate: the app runs locally against a DEMO mailbox that automatically copies e-mails from the production mailbox, so the sender is frequently NOT azuredevops@microsoft.com. The decision: categorization must gate on FOLDER MEMBERSHIP only and derive the triple purely from body signals (the ADO URL + action sentence); sender is never a gate. This matters because the story text and the domain rule still describe the native-sender check as central, but the actual runtime/deployment reality (a copy-to-demo-mailbox flow, adopted because the primary mailbox can't be connected locally for security reasons) makes any sender-based gating actively wrong. The cause is that the domain rule encodes an assumption about sender identity that the project's real mail-delivery topology contradicts, and that topology is not recorded anywhere in the config.
