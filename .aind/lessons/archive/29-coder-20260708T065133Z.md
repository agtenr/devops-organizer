## Observation
Scaffolding against the current toolchain (TypeScript 6, ESLint 10, Vite 8, Vitest 4) surfaced three version-specific gotchas the plan did not anticipate, all of which will recur for every future story that touches this config:
1. TypeScript 6 rejects `noEmit` on a *referenced* composite project (TS6310) when the referencing project is itself composite, so the plan's two-file layout (tsconfig.json + tsconfig.node.json) does not build with `tsc -b`. The working shape is the canonical solution-root layout: a `tsconfig.json` with `files: []` + references, plus a separate `tsconfig.app.json` and `tsconfig.node.json`.
2. eslint-plugin-react-hooks `configs['recommended-latest']` ships its `plugins` field as an array, which ESLint 10 flat config rejects (flat config requires `plugins` to be an object); the config had to register the plugin manually and spread only its `rules`.
3. Vitest's default test glob picks up Playwright specs under `e2e/`, causing "test() called here" failures; Vitest needs an explicit `include: ['src/**/*.{test,spec}.{ts,tsx}']` to stay out of the Playwright directory when both runners coexist.
The cause is that the plan/rules were written before the toolchain existed and assumed older-version conventions; the pinned bleeding-edge versions changed the required config shapes.
