## Observation
The plan called for pruning the selection to the currently-visible rows on a filter change. The natural implementation — a useEffect that setState-prunes when the visible-id set changes — was rejected by this project's ESLint config, which enforces the newer react-hooks rule `react-hooks/set-state-in-effect` ("Avoid calling setState() directly within an effect"). The fix was to derive the effective selection during render (intersect raw picks with visible ids via useMemo) instead of syncing state in an effect. Why it matters: frontend-architecture.md documents "ESLint + react-hooks rules" generally but not that this stricter set-state-in-effect rule is active, so a plan/coder that reaches for effect-based state syncing will hit a lint failure late (at the build/lint gate) rather than being steered to derived state up front.
