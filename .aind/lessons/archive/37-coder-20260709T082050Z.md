## Observation
npm run format:check fails on a Windows checkout for the entire repo — including files untouched by the change — because core.autocrlf=true checks files out as CRLF, there is no committed .gitattributes, and Prettier defaults to endOfLine "lf", so every file reads as a violation. This is exactly the false-positive that frontend-architecture.md documents and prescribes a fix for (.gitattributes eol=lf and/or Prettier endOfLine), yet the repo still has neither, so the fix was never actually applied. The friction recurs for every contributor on Windows and forces per-change workarounds (checking only the changed files in isolation); it likely also silently passed in prior stories only because those commits happened on a non-CRLF checkout with no CI to catch it. The cause is a documented convention that was never committed to the repo.
