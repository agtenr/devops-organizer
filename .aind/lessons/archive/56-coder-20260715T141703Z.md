## Observation
On a fresh implement worktree (branched off origin/main), `npm run test` had two failing suites that had nothing to do with story 56: (1) categorizationService.test.ts — expected, since the .msg corpus and generated fixtures are both gitignored, so a clean checkout has no fixtures to load; and (2) Organizer.test.tsx "renders the tabs, filters, and list on success", which fails IDENTICALLY on the base commit with the branch changes stashed — i.e. a red test is sitting on the integration branch. The project has no CI gate (CLAUDE.md: "CI/CD: none configured yet"), so a test can be (and was) merged while failing, and every downstream build phase inherits a non-green baseline. This makes the recurring DoD line "npm run test passes" unverifiable in a clean environment and forces each coder to manually separate "my failures" from "pre-existing/environment failures" by stashing and re-running.
