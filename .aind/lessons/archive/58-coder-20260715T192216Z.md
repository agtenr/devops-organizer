## Observation
The plan named the pure derivation helper `selectedFilters.ts`, colocated next to the component `SelectedFilters.tsx`. These filenames differ ONLY by the case of the first letter, which collides on case-insensitive filesystems (Windows dev machines): `import { SelectedFilters } from './SelectedFilters'` resolved to the helper module (which has no `SelectedFilters` export), so the component was `undefined` at runtime and every render-based test failed with "Element type is invalid ... got: undefined". Cause: the frontend-architecture convention "a component's pure helper gets its own colocated file named for its content" naturally tempts a helper name that is the camelCase of the PascalCase component (SelectedFilters -> selectedFilters), and nothing in the rule warns that a helper filename must differ from its sibling component filename by more than letter case. Existing pairs avoid this by chance (SidebarFilters/facetFilters, EmailList/emailFormatters) because their helper names are semantically different words, not the camelCase of the component. Renaming to filterChips.ts fixed it. Worth capturing because it will recur whenever a new component's helper is named the camelCase of the component.
