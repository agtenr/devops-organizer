## Observation
Planning this UI story surfaced a recurring coupling in the mock-data harness seam: `src/harness.tsx` hand-maintains a full literal `OrganizerData` object (`mockData`), and `OrganizerData` is `ReturnType<typeof useOrganizer>`. So *any* story that adds a field to `useOrganizer`'s return (here `selectedFilters` + `removeFilter`) silently breaks the harness build until the mock is updated in lockstep — and because the screenshot rule mandates that every UI story drive its evidence through this same hand-written mock, each such story must also seed new selection state there by hand. The cause is that the seam's mock is a manually-kept parallel of the real hook's shape with no shared factory/default, so shape drift is only caught at compile time in the harness, not guided by the rules. Worth capturing because it recurs on every data-shape-changing UI story, not just this one.
