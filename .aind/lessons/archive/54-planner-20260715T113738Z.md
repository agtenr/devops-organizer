## Observation
For resizable columns on the e-mail list, the planner recommended the minimal-diff approach (add useTableColumnSizing_unstable to the existing primitive Fluent Table, preserving the custom selection and row-click behaviour). The human reviewer instead chose the idiomatic full DataGrid migration (O1-B) despite it being the larger, higher-risk change. Signal: for this project's Fluent v9 UI, the human weights using the idiomatic/first-class component (DataGrid for a resizable grid) over minimising diff size and risk — the "simplest change" bias did not match the reviewer's preference here. A secondary consequence worth noting: choosing DataGrid then collided with an already-accepted narrower decision (O2-A: only data columns resizable), because DataGrid's resizableColumns is global with no per-column opt-out — a reminder that framework-level feature choices can retroactively invalidate finer-grained scoping decisions.
