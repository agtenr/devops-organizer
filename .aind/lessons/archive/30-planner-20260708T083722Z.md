## Observation
While planning the authentication-setup story, the authentication rule's "least privilege — read-only mail access (Mail.Read)" invariant read as if every auth change should request Mail.Read. But sign-in setup needs no Graph scope at all: the display name comes from the ID-token 'name' claim, and Mail.Read only becomes relevant in the later mail-reading story. The rule conflates the app's eventual scope with what the auth-setup slice needs, which could push an implementer to over-request scopes (broader than least-privilege) before any mail is read. Cause: the rule states the end-state scope as an always-on invariant without distinguishing sign-in-only from mail-reading stages.
