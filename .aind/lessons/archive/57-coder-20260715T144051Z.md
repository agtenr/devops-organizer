## Observation
Resolving a PR merge conflict in revise mode requires rebasing the feature branch onto the moved integration branch, which rewrites history — but the aind-revise-code-pr.sh `push` subcommand does a plain `git push`, which is rejected as non-fast-forward after a rebase. The coder had to fall back to a manual `git push --force-with-lease`. The friction: the standard revise "push" step assumes a fast-forward and has no path for the (common) case where the human's directive was "fix the conflicts" and the fix is a rebase.
