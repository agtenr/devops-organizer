## Observation
The story's §7 reference table lists friendly project NAMES for all 14 sample e-mails, but building against the real /design/demo-messages corpus revealed that Azure DevOps work-item and release notifications carry only the project GUID in their URLs — the friendly name is genuinely absent from 6 of the 14 e-mails. So the plan's Definition-of-done "the 14 demo messages match the §7 reference-examples table" is not literally achievable in a front-end-only app (no GUID→name lookup source), and the engine can only output the GUID for those 6. Separately, the merged plan narrowed AC §2.5 to the no-URL case and defined needsReview as body-signal-only, but never reconciled AC §6's explicit instruction that an "untranslatable GUID" be flagged for review — so a plan-faithful implementation contradicts the literal story AC. Both gaps were invisible at plan time and only surfaced when implementing against real sample data. The cause: the story's reference table and §2.5/§6 rules were written from human knowledge of the orgs (mentally translating GUIDs), an assumption the actual e-mail payloads don't support, and the plan didn't catch the mismatch because the real corpus wasn't inspected during planning.
