## Observation
I planned to keep login OIDC-only and request Mail.Read incrementally at first fetch (via AuthCodeMSALBrowserAuthenticationProvider with a redirect consent), following authentication.md's scope-staging note that "Mail.Read is requested only when mail is first read." The reviewer overrode this: fold Mail.Read into the sign-in request so the user consents once, then acquire tokens silently (acquireTokenSilent) — explicit priority was minimizing user interaction ("I don't want to involve the user too much"), which the incremental-consent-at-fetch redirect violated. So the rule's least-privilege-staging preference (defer the mail scope until first read) conflicted with the reviewer's stronger UX preference (one consent at login, silent thereafter). Cause: the rule framed scope timing purely as least-privilege staging and did not weigh the user-interaction cost of deferring consent to a mid-session redirect.
