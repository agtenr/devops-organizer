## Observation
The frontend-architecture rule documents a "line endings pinned to LF deterministically" convention and even names the fix files (.gitattributes, .prettierrc endOfLine), but the repo had not actually implemented it: .gitattributes was absent and .prettierrc had no endOfLine. With core.autocrlf=true on a Windows checkout, the working tree is CRLF and Prettier's default endOfLine:lf flags EVERY file (incl. untouched ones), so `npm run format:check` — a Definition-of-done bar cited by the rule — fails wholesale for any Windows contributor. Cause: a documented convention was left un-applied in the codebase, so the rule and the actual repo state diverged and the DoD's format:check gate was unmeetable on Windows until the coder added the prescribed .gitattributes/.prettierrc.
