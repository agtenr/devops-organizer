## Observation
The human ran the app locally and found three defects that the full green test suite missed: clicking a row opened the body panel at ~0px width (invisible InlineDrawer, flex-shrunk by the flex-grow list), the table columns split evenly instead of sizing Subject widest, and the needs-review badge wrapped to two lines. Root cause of the escape: the project's test practice (testing.md) mandates jsdom component tests, which have no layout/CSS engine — an element that is "open" in the DOM but 0px wide, or columns with wrong widths, still satisfy DOM/role/text assertions, so the component test asserting the panel opened passed while the panel was invisible in a real browser. The human's reaction ("shouldn't the test fail? this cannot happen in the future") is the signal: interactive/visual UI behaviour is not covered by jsdom component tests alone, yet neither testing.md nor the /implement self-check requires exercising the rendered UI in a real browser before a UI story is called done.
