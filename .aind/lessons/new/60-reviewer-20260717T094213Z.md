## Observation
Making a heading interactive by rendering a Fluent Button with role="heading"/aria-level (the pattern OQ1 option A approved for the app title) preserves the h1 landmark but overrides the native button role, so assistive tech announces the element as a heading, not as a clickable control — AT users get no cue that activating it refreshes the page. The tension recurs whenever a title/heading must also be a clickable action: the ARIA role override trades away the interactive affordance to keep the landmark. The more accessible shape (a heading element wrapping a button, e.g. <h1><button>, as Fluent's own Accordion header uses) keeps both semantics, but the "Button + role=heading" phrasing in the plan actively steers toward the role-override that loses the button announcement.
