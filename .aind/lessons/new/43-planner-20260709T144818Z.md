## Observation
authentication.md keeps a per-story "Scope staging" log and asserts "never broaden beyond read-only mail access", but it gives no guidance on WHO updates that log when a feature story legitimately introduces a new scope. Story 43 (delete mail) needs Mail.ReadWrite, which will make the rule's standing claim false the moment the code ships, yet nothing says whether a feature-implementation PR should edit the rule doc or whether rule-doc maintenance is reserved for the human/dreaming phase. This forced an open question that is really a process gap in the rule itself: a living per-story log embedded in a rule doc has no defined update owner, so every scope-changing story will re-raise the same ambiguity.
