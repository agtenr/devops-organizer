## Observation
The testing rules position the mock-data harness (/harness.html) as THE seam for real-browser (Playwright) verification of interactive acceptance. But the harness models filter state via URL params (?state=filtered) rather than live React state, and its selection callbacks are no-ops (src/harness.tsx). So an acceptance criterion whose behaviour depends on live in-memory state transitions — here, "click the title -> window.location.reload() -> in-memory filters clear" — cannot be faithfully driven through the harness: reload() preserves the query string, so a harness reload re-seeds the URL-encoded filters instead of clearing them, and the harness has no interactive filter state to reset. The result is a genuine gap: an interactive AC that the rules imply the harness can E2E, but that in practice only construction-reasoning + a component unit test + a manual live check can verify. The cause is the harness being a static/URL-driven stand-in rather than a live-state mount, which the testing rule does not flag as a limit on what it can verify.
